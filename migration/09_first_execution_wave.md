# Первая Исполнительная Очередь

## Цель Первой Очереди

Первая очередь должна превратить foundation в короткий рабочий пакет для первого реального переноса.

Она должна дать:
- зафиксированный первый migration unit;
- его target contract;
- его parity surface;
- минимальную рабочую структуру внутри `wb-core` для дальнейшей реализации.

Она не должна пытаться:
- переносить business-модули;
- писать production-код;
- строить новую таблицу;
- закрывать весь roadmap заранее.

## Что Делаем В Первой Очереди

Порядок:
- окончательно зафиксировать первый migration unit: `web-source snapshot block`;
- описать его target contract внутри `wb-core` на уровне входов, выходов, failure-mode и границ ответственности;
- зафиксировать parity matrix для этого блока относительно текущего legacy contract;
- определить минимальный набор fixtures/evidence, который понадобится до начала реализации;
- подготовить базовую repo-структуру под этот блок внутри `wb-core` без business-логики и без runtime-реализаций.

## Что Не Делаем В Первой Очереди

Не делаем:
- реализацию API, jobs, ingestion или browser capture;
- перенос `AI_EXPORT`, `CONFIG`, `METRICS` или других legacy-контуров целиком;
- новую operator-table;
- cutover;
- рефакторинг legacy;
- длинный multi-module plan.

## Почему Именно Такой Порядок

Сначала фиксируется один bounded unit.

Потом фиксируется его contract и parity surface.

Только после этого имеет смысл готовить repo-slice под реализацию.

Иначе снова получится смешение архитектуры, runtime и миграции без жёсткой границы.

## Критерии Завершения Первой Очереди

Первая очередь закрыта, если:
- первый migration unit зафиксирован без двусмысленности;
- target contract для него описан в `wb-core`;
- parity matrix и required evidence перечислены явно;
- ясно, какие артефакты и тестовые поверхности нужны до начала реализации;
- внутри `wb-core` подготовлена минимальная структура под этот блок без production-кода.

## Первый Следующий Технический Шаг

Первый следующий технический шаг:
- создать в `wb-core` короткий contract-first документ для `web-source snapshot block`, который фиксирует target contract, границы, parity surface и required evidence для будущей реализации.
